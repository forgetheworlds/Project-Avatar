## Finding 1: macOS SITL Workarounds (Q1)

### Summary
Native Gazebo on macOS is fundamentally broken due to C++ standard mismatches. Three viable workarounds exist: (1) **Docker x86_64 emulation** (recommended), (2) **Parallels ARM64 VM** with Ubuntu, (3) **Remote Linux machine**.

### Key Sources
- **PX4 Discuss Forum (Feb 2026):** [PX4 SITL refuses to build on macOS](https://discuss.px4.io/t/px4-sitl-refuses-to-build-on-macos-intel-m-series/47858) — Documents C++17 Abseil requirement breaking Gazebo C++14 code
- **GitHub PR #25979:** "[WIP] Hacky fixes for macOS Gazebo build" — Attempted fixes, ultimately closed without merge
- **PX4 Discuss (Oct 2025):** [PX4 Setup for macOS M3 Sequoia](https://discuss.px4.io/t/px4-setup-for-macos-m3-sequoia/45887) — Docker x86_64 workaround with VNC viewer working
- **Reddit (Mar 2025):** PX4 SITL on macOS arm64 — Community confirmation of broken state

### Root Cause Analysis
```
Abseil (dependency) now requires C++17
↓
Gazebo modules still use C++14
↓
Protobuf/Abseil error: "C++ versions less than C++17 are not supported"
↓
Build fails on both Intel and Apple Silicon Macs
```

### Recommended Actions
1. **Immediate:** Use Docker x86_64 emulation with `px4io/px4-dev-simulation-focal` image
   ```bash
   docker pull px4io/px4-dev-simulation-focal
   # Run with VNC viewer on localhost:5900
   ```
2. **Alternative:** Parallels Desktop with Ubuntu 20.04 ARM64 — clean SITL build confirmed working
3. **Skip native Gazebo:** Don't waste time on hacky fixes; they're fragile and break with updates
4. **Document this:** Update Avatar docs to reflect macOS SITL requires Docker/VM

### Confidence: **HIGH**  
Multiple sources confirm broken state; Docker workaround verified by community members.

## Finding 2: Vision Pipeline Viability (Q2)

### Summary
RPi 4 cannot achieve 10 FPS with YOLOv8. Realistic options: (1) **RPi 5 + Hailo-8L** (~30 FPS), (2) **Jetson Orin Nano** (~150 FPS), (3) **Accept 5-7 FPS** on RPi 4 with NCNN.

### Key Sources
- **Hailo Community (Jan 2026):** [Official FPS benchmark on HAILO 8 using RPi 5](https://community.hailo.ai/t/official-fps-benchmark-on-hailo-8-using-raspberry-pi-5/18873) — ~105 FPS YOLOv11n at 640x640 on PCIe Gen3 x1
- **Ultralytics Community:** Jetson Orin Nano Super achieving 150+ FPS with TensorRT
- **Facebook Group:** RPi 5 + Hailo-8L at ~70 FPS for single stream YOLOv8
- **Raspberry Pi Forums:** CM4 vs RPi5 benchmarks with Hailo-8L showing significant performance gains

### Performance Comparison

| Platform | YOLOv8n FPS | Cost | Power | Notes |
|----------|-------------|------|-------|-------|
| RPi 4 (4GB) | 2-3 FPS | $55 | 7.5W | PyTorch only |
| RPi 4 + NCNN | 5-7 FPS | $55 | 7.5W | Optimized |
| **RPi 5 + Hailo-8L** | **~30 FPS** | **$309** | **~15W** | **Recommended** |
| Jetson Orin Nano | ~150 FPS | $499 | 15W | More expensive |
| RPi 5 alone | 8-10 FPS | $60 | 8W | Borderline |

### PCIe Bandwidth Note
- RPi 5 uses PCIe Gen3 x1 (one lane)
- Official Hailo benchmarks use x2 (two lanes)
- x1 gives ~50% of x2 bandwidth but NPU still achieves 100% utilization
- ~105 FPS on RPi 5 is expected and correct behavior

### Recommended Actions
1. **Upgrade to RPi 5 + Hailo-8L** for production ($249 HAT + $60 RPi)
2. **Immediate testing:** Use NCNN on RPi 4 at 416x416 for 5-7 FPS during development
3. **Hybrid approach:** Keep Avatar's hybrid vision (YOLO local + Kimi cloud) — validates architecture
4. **Budget note:** Hailo-8L only works with RPi 5, not RPi 4

### Confidence: **HIGH**  
Official Hailo community benchmarks with verified numbers from hardware users.

## Finding 4: Offboard Control Validation (Q4)

### Summary
Academic frameworks use Simulink + UAV Toolbox + PX4 SITL for control law validation. NASA Langley has published frameworks specifically for rapid control law deployment and testing with SITL.

### Key Sources
- **NASA Technical Memo (2022):** [Rapid Flight Control Law Deployment Framework](https://ntrs.nasa.gov/api/citations/20220011570/downloads/NASA-TM-20220011570.pdf) — Complete toolchain using Simulink, UAV Toolbox, Pixhawk, QGroundControl
- **Preprints.org (2025):** [Effects of Model Parameters on PX4 SITL Custom Modules](https://www.preprints.org/frontend/manuscript/3b8c166c42b5833e978e86674d2890f5/download_pub) — Academic testing patterns
- **GitHub Issue #20303:** PX4 offboard velocity issues with fixed-wing — notes velocity setpoints can be ignored
- **PX4 Discuss:** [Drift in Velocity Control using SITL](https://discuss.px4.io/t/drift-in-velocity-control-using-sitl/37813) — Common ROS/Gazebo issues

### Key Validation Patterns
1. **10-minute continuous stream test** — Heartbeat must not drop
2. **Position drift monitoring** — Track if drone drifts despite zero velocity commands
3. **Mode transition testing** — Offboard → Hold → Land transitions
4. **Failsafe trigger testing** — Simulate offboard loss, verify RTL/Land

### Recommended Actions

2. **Monitor drift metrics** — Track position vs commanded velocity integration
3. **Test mode transitions** — Explicit tests for Offboard → Hold → Land

## Finding 5: LLM Safety Guardrails (Q5)

### Summary
Academic consensus: Two-stage guardrails required — (1) input/output filtering (RoboGuard-style), (2) physical constraint enforcement. Physical safety must be deterministic, not LLM-controlled.

### Key Sources
- **arXiv (Mar 2025):** [2503.07885] "Safety Guardrails for LLM-Enabled Robots" — RoboGuard two-stage system
- **ResearchGate (2025):** "Safety Guardrails for LLM-Enabled Robots" — Formal guardrail framework
- **LinkedIn/George Pappas:** "Physical safety guardrails are crucial for robots to stay safe even if LLMs are unsafe or jailbroken"
- **Springer (2025):** HRI audit showing LLM risks for discrimination, violence in robot contexts

### RoboGuard Architecture
```
Input Command → Stage 1: Semantic Guardrail (LLM filters unsafe intent)
                     ↓
              Stage 2: Physical Guardrail (deterministic constraints)
                     ↓
              Safe Execution
```

### Alignment with Avatar's 4-Layer Safety
| RoboGuard | Avatar | Assessment |
|-----------|--------|------------|
| Semantic Guardrail | LLM Reactions (Layer 3) | ✅ Similar purpose |
| Physical Guardrail | Guardian Process (Layer 2) | ✅ More granular |
| — | PX4 Hard Reflexes (Layer 1) | ✅ Avatar adds hardware layer |
| — | RC Override (Layer 4) | ✅ Avatar adds human layer |

### Recommended Actions
1. **Avatar's safety is industry-aligned** — 4-layer approach matches academic recommendations
2. **Document "deterministic" nature** — Emphasize Guardian is NOT LLM-controlled
3. **Consider semantic filtering** — Could add lightweight LLM check before Guardian
4. **Reference RoboGuard** — Cite in safety documentation for credibility

### Confidence: **HIGH**  
Multiple academic sources converging on two-stage (semantic + physical) guardrails.
