## Finding 1: DroneServer — Battle-Tested MCP Architecture with Hard-Won Safety Lessons

### Summary

**DroneServer (PeterJBurke/droneserver) is a 3,350-line Python MCP server that lets any MCP-compatible LLM control real drones via MAVLink.** It has 41 MCP tools across 6 categories, was tested on a real sub-250g quad with Raspberry Pi Zero W bridge, and is production-quality open source. The most valuable artifact for Avatar is NOT the MAVLink code but the **safety architecture**: DroneServer actively manages the flight lifecycle — it is NOT a simple command pass-through. Key safety mechanisms include: Takeoff Altitude Wait (blocks navigation until altitude reached), Landing Gate (blocks landing if >20m from registered destination), Auto-Land with Confirmed Touchdown (monitor_flight waits for physical ON_GROUND + stable for 3 seconds), and 30-second mandatory progress updates. These patterns are directly portable to Avatar's UDP-based MCP server.

**The crash report (Nov 2025) is the most educational document:** `pause_mission()` caused a real drone crash from 25m because LOITER mode in ArduPilot requires RC throttle at 50% center to hold altitude — without a physical RC controller, the drone descended at 5 m/s to ground impact. This is exactly the kind of integration bug Avatar will face with custom firmware. The lesson: **every flight mode transition must be verified for altitude behavior before deploying to hardware.**

### Key Sources

- **DroneServer repo** (github.com/PeterJBurke/droneserver, cloned and analyzed): 3,350 lines Python, 41 MCP tools, MAVSDK-Python + FastMCP + asyncio. Tool registration pattern: `@mcp.tool()` decorator on async functions. Global `MAVLinkConnector` dataclass with `connection_ready` asyncio.Event for connection management. `pending_destination` tracking for landing gate. `landing_in_progress` flag for monitor_flight coordination.
- **STATUS.md** (in-repo): Documents v1.4.0 "Complete Flight Lifecycle Management" — production-ready with confirmed touchdown detection. The LLM cannot return `mission_complete: true` until the drone is physically on the ground.
- **LOITER_MODE_CRASH_REPORT.md** (in-repo): Full post-mortem of the Nov 2025 crash. Root cause: LOITER mode uses RC throttle for altitude — without a physical RC controller at 50% center, altitude becomes unpredictable. Fix: deprecated `pause_mission()`, replaced with `hold_mission_position()` that stays in GUIDED mode.
- **CHATGPT_SETUP.md** (in-repo): Recommended prompt pattern forces the LLM to call `monitor_flight` "at least 20 times or until mission_complete is true" — prevents LLM from giving up mid-mission.

### Tool Architecture Deep-Dive

```
DroneServer Tool Categories (41 tools total):
├── Flight Control (5): arm_drone, disarm_drone, takeoff, land, hold_position
├── Emergency & Safety (3): return_to_launch, kill_motors, get_battery
├── Navigation (8): get_position, go_to_location, monitor_flight, set_yaw,
│                    move_to_relative, reposition, check_arrival, fly_to
├── Mission Management (10): initiate_mission, upload_mission, start_mission,
│                            hold_mission_position, resume_mission, etc.
├── Telemetry (12): get_health, get_health_all_ok, get_landed_state,
│                   get_heading, get_rc_status, get_odometry, get_imu, etc.
└── Parameter Management (3): get_parameter, set_parameter, list_parameters
```

**Key code patterns Avatar should adopt:**

1. **Connection Gate Pattern** — Every tool calls `ensure_connection(connector)` first, blocking until the drone connection is ready. For Avatar UDP: every tool should verify the ESP32 is reachable before sending commands.

2. **Landing Gate Pattern** — `land()` checks `pending_destination` (set by `go_to_location()`) and calculates haversine distance. If >20m from destination, landing is BLOCKED with: `"Cannot land - drone is Xm from destination!"`. This prevents the LLM from landing early. Avatar should implement the same.

3. **Takeoff Altitude Wait** — `takeoff(altitude=3.0, wait_for_altitude=True)` polls telemetry at 1Hz until within 0.5m of target, with 60s timeout. Returns `safe_to_navigate: True` only when altitude is reached. Critical for preventing navigation at low altitude.

4. **monitor_flight Pattern** — Returns progress every 30s (hardcoded, LLM cannot override). Each response includes `DISPLAY_TO_USER` string and `mission_complete` boolean. Auto-lands when within 20m of destination, then waits for ON_GROUND + stable for 3 seconds. **This is the most important pattern for Avatar**: the LLM generates the plan upfront, a simpler execution engine runs it, and the LLM only intervenes for exceptions.

5. **Global Connection Singleton** — `_global_connector` with `asyncio.Lock` ensures all MCP tool calls share one persistent MAVLink connection. For Avatar UDP: maintain one persistent UDP socket, not create/destroy per command.

### Recommended Actions

- **IMMEDIATE:** Read the LOITER_MODE_CRASH_REPORT.md in full. It's an excellent example of the class of bugs Avatar will encounter — every flight mode in the custom firmware must be tested for altitude behavior.
- **THIS WEEK:** Design Avatar's MCP tool set copying DroneServer's 6-category structure. Start with: `arm()`, `takeoff(altitude)`, `goto(lat, lon, alt)`, `orbit(center, radius, speed, height)`, `follow_subject(description)`, `land()`, `monitor_flight()`. Each tool should have the same safety gates.
- **THIS MONTH:** Implement a minimal MCP server (FastMCP Python) that sends UDP commands to ESP32. The tool functions follow DroneServer's exact pattern: `@mcp.tool() async def takeoff(ctx, altitude) -> dict:` → validate → send UDP → wait for telemetry confirmation → return result.
- **CRITICAL:** The `monitor_flight` pattern is the solution to the LLM "getting bored" problem DroneServer documented (LLMs stop calling tools after 5-10 min). Avatar's monitor_flight should: send waypoints in batches of 20, wait for each to complete, provide progress, auto-land on completion, keep LLM engaged via `action_required` field.
- **NOTE:** DroneServer's architecture (MAVSDK-Python + FastMCP) runs on a separate machine from the drone. This matches Avatar's architecture exactly — MCP server on MacBook, UDP commands to ESP32.

### Confidence: **HIGH**

## Finding 2: madflight — Production-Grade Modular FC Architecture vs LiteWing's Turnkey Approach

### Summary

**madflight (qqqlab/madflight) is a production-grade, multi-platform flight controller library** supporting ESP32-S3, ESP32, RP2350, RP2040, and STM32. It has 551 commits, 30 releases, 418 GitHub stars, MIT license, and commercial hardware available on Tindie. The architecture is modular: separate pluggable "gizmo" drivers for each IMU (MPU6000/6500/9250, BMI270, ICM42688/42688P, ICM45686), GPS (u-blox), optical flow (PMW3901), rangefinder (VL53L3CX, LD2411S, SR04), magnetometer (RM3100, BMM150). It supports PWM, Oneshot125, DShot300 (bidir), and brushed motor outputs. MAVLink v2 native. The PID architecture is a clean C++ class with P/I/D, integrator saturation limits, derivative-on-measurement option, and 360-degree yaw wrapping.

**madflight vs LiteWing comparison:**

| Aspect | madflight | LiteWing |
|--------|-----------|----------|
| **Platforms** | ESP32-S3/ESP32/RP2350/RP2040/STM32 | ESP32-S3 only |
| **IMU support** | MPU6xxx, BMI270, ICM42688/P, ICM45686 (auto-detect) | MPU6050 only (hardcoded in ESP-Drone firmware) |
| **Motor outputs** | PWM, Oneshot125, DShot300, DShot300 bidir, Brushed | Brushed MOSFET PWM only |
| **RC protocols** | CRSF, ELRS, SBUS, IBUS, DSM, MAVLink | WiFi phone control + BLE gamepad |
| **Sensor fusion** | AHRS (Mahony/Madgwick) + GPS + optical flow | Mahony filter + optical flow positioning |
| **MAVLink** | Native v2 with full library | Not supported (uses custom UDP protocol) |
| **CLI** | Built-in MSP protocol + serial CLI | Custom serial commands |
| **GPS** | u-blox binary protocol (high-rate) | Not supported (positioning relies on optical flow) |
| **Maturity** | 551 commits, 30 releases, active maintenance | 63 commits (newer), Kickstarter-funded |
| **License** | MIT | Open hardware (exact license unclear from repo) |
| **Code quality** | Professional C++ with abstract gizmo pattern | Functional but monolithic ESP-IDF C code |
| **Documentation** | madflight.com website + source comments | Circuit Digest tutorials + wiki |
| **Hardware** | Reference designs, Tindie FC available | Kickstarter kit ($69), custom PCB |
| **PID architecture** | Clean PIDController class with deriv-on-measurement | C-style PID in main loop |
| **Learning curve** | Moderate (needs configuration) | Low (pre-configured for known hardware) |

**The motor output module is a decisive advantage for Avatar:** madflight's `out.cpp` supports `setup_brushed(4, motor_outputs, 5000)` — 5kHz PWM, 0-100% duty cycle, exactly what Avatar's coreless motors need. The higher PWM frequency (5kHz vs typical 400Hz) eliminates audible motor whine, which is important for cinematic use (no motor noise in audio). LiteWing uses basic MOSFET on/off PWM at lower frequencies.

### architecture comparison: imu_loop()

**madflight quadcopter example (10.Quadcopter/main.cpp):**
```cpp
void imu_loop() {                    // Called on IMU interrupt
    led_Blink();                     // Visual status
    ahr.update();                    // Mahony/Madgwick sensor fusion
    control_Rate(rcl.throttle == 0); // PID: setpoint → motor commands
    out_KillSwitchAndFailsafe();     // Safety: disarm on RC loss
    out_Mixer();                     // Quad mixing + PWM output
}
```
The user writes ONLY 370 lines — madflight handles IMU drivers, AHRS, RC protocols, CLI, MSP, logging, and FreeRTOS task scheduling.

## Finding 3: M10Q GPS on 1S LiPo — Brownout Risk Is Real, Protection Is Cheap

### Summary

The SAM-M10Q GPS module operates at V_IO = 2.7V to 3.6V (when VIO_SEL=open). On a 1S LiPo (4.2V fully charged, sagging to ~3.3V under 4×8520 motor bursts at ~3A total), the GPS is at risk of brownout if powered through an LDO with insufficient dropout margin. **The critical finding: a standard 3.3V LDO powered from 1S has only ~0.3-0.7V dropout margin during motor bursts — dangerously close to brownout.** The fix is simple and cheap: add a 100-470µF bulk capacitor at the GPS power input, and optionally use a dedicated low-dropout regulator (e.g., AP2112K-3.3) with <250mV dropout. The backup domain (V_BCKP, 1.65V min) should be powered by a small supercapacitor or Li battery to maintain hot-start capability.

**Cold start: 23-28 seconds** (4 GNSS constellations). Hot start: 1 second. Assisted (AssistNow): 1-2 seconds. This means first-flight GPS lock after power-on takes ~30 seconds — acceptable. But if GPS browns out during flight and performs a cold restart, the drone loses position for 23+ seconds — unacceptable for cinematic flight (position drift would ruin the shot).

### Key Sources

- **SAM-M10Q Datasheet** (UBX-22013293, u-blox): V_IO = 2.7V to 3.6V, V_BCKP = 1.65V to 3.6V. Cold start 23s (GPS+GAL+BDS+GLO), Hot start 1s. Max navigation update rate: 10 Hz (default 4 concurrent), 20 Hz (high performance mode). Position accuracy: 1.5m CEP. Sensitivity: -165 dBm tracking, -146 dBm cold start.
- **MIA-M10Q Datasheet** (UBX-22015849, u-blox): V_IO = 1.76V to 1.98V (VIO_SEL=GND) or 2.7V to 3.6V (VIO_SEL=open). ICC_RF = 50mA max. Leakage current: 25nA. Operating temp: -40°C to +85°C. The chip itself is robust — the brownout risk is in the power supply, not the GPS chip.
- **SparkFun SAM-M10Q Breakout** (sparkfun.com): Includes 3V/1mAh lithium battery for V_BCKP backup domain. This battery maintains the GNSS ephemeris and almanac data, enabling hot starts within seconds after brief power interruptions. Avatar should replicate this on the custom PCB.
- **NewBeeDrone M10Q** (newbeedrone.com, 2.16g): Designed specifically for micro FPV drones. Likely includes onboard power filtering given the target use case. At 2.16g, it's the lightest GPS+compass module. This should be Avatar's GPS.
- **Community knowledge** (Reddit r/diydrones, r/esp32): The brownout prevention checklist from the Apr 28 discovery applies here: (1) large electrolytic cap (100-470µF) across battery input, (2) dedicated 3.3V LDO with ≥500mA rating, (3) separate motor/logic power domains with Schottky diode OR gate, (4) 0.1µF ceramic caps at each MOSFET.

### Brownout Risk Calculation

```
1S LiHV nominal:          4.2V (charged) → 3.7V (nominal) → 3.3V (cutoff)
4× 8520 motors at full:   ~3A draw at 3.7V
Battery internal resistance: ~50-100mΩ (1S 380mAh)
Voltage sag at 3A:        3A × 0.1Ω = 0.3V drop at battery terminals
Battery voltage under load: 3.7V - 0.3V = 3.4V
LDO dropout (AP2112K):    ~250mV at 300mA
LDO output:               3.4V - 0.25V = 3.15V — BELOW M10Q minimum (3.3V spec, but MIA-M10Q shows V_IO can go to 2.7V at reduced performance)
```

**The MIA-M10Q variant (used in NewBeeDrone) supports V_IO down to 2.7V** — this provides a margin of safety. The SAM-M10Q datasheet also shows V_IO minimum 2.7V. So the actual brownout threshold is ~2.7V, not 3.3V. With 3.15V under worst-case sag, there's ~450mV of margin. **Brownout risk is LOW for M10Q modules with 2.7V minimum V_IO. But still add the bulk capacitor for safety.**

### Recommended Actions

- **PCB DESIGN:** Add a 100µF tantalum or 220µF electrolytic capacitor at the GPS 3.3V rail. Place it physically close to the GPS module. Cost: <$0.50, weight: <1g.
- **PCB DESIGN:** Include a 0.1F supercapacitor or CR1220 battery holder for the V_BCKP backup domain. This enables hot starts after brief power glitches. The NewBeeDrone M10Q likely has this built in — verify.
- **PCB DESIGN:** Use a dedicated 3.3V LDO with <300mV dropout for the GPS. Do NOT share the LDO with the ESP32-S3 (which draws 200-500mA). GPS draws <50mA — use a small SOT-23 LDO.
- **FIRMWARE:** Monitor GPS fix status continuously. If fix is lost for >2 seconds during flight, immediately: (1) freeze position (hold last known position via optical flow), (2) if optical flow also unavailable, initiate auto-land, (3) log the brownout event for post-flight analysis.
- **TESTING:** Before first flight, run a ground test: spin all 4 motors at full throttle while monitoring GPS output on serial. Verify no NMEA sentence drops or fix losses. This is a 10-minute test that could prevent a 23-second position blackout during flight.

### Confidence: **MEDIUM-HIGH**
Datasheet voltage specs are definitive. The brownout margin calculation is conservative but realistic. The untested variable is the actual voltage sag of the specific 1S 380mAh LiHV battery under 8520 motor load — this should be measured with a scope during ground testing.

## Finding 5: External IMU Sync for Gyroflow — Feasible But RunCam Thumb Pro Has Known Sync Issues

### Summary

**Avatar can use the ESP32's ICM-42688 IMU for Gyroflow stabilization instead of relying on the RunCam Thumb Pro's internal gyro.** The Gyroflow `.gcsv` format is well-documented and easy to generate from ESP32. The ICM-42688 produces superior data (less noise, lower drift) compared to the budget IMU in the RunCam. However, **the RunCam Thumb Pro has documented sync issues** (duplicated/dropped frames, unreliable sync point colors) — Gyroflow's own documentation recommends avoiding RunCam cameras for reliable gyro sync. This doesn't mean Avatar can't use the Thumb Pro — it means Avatar should NOT rely on the Thumb Pro's internal `.gcsv` file. Instead, log gyro data from the ESP32's ICM-42688 and sync manually with the video.

**The sync method:** Start recording on both ESP32 gyro log and RunCam video simultaneously. In post, Gyroflow's autosync algorithm uses optical flow from video frames to match against gyro data — it will find the offset automatically. For manual sync (if autosync fails): perform a rapid attitude change (quick pitch/roll) visible in both the video and gyro data as a sharp spike — this becomes the manual sync point.

### Key Sources

- **Gyroflow .gcsv Format** (docs.gyroflow.xyz): Text-based sidecar file. Header: `GYROFLOW IMU LOG` + `version,1.3` + metadata + scaling constants (tscale, gscale, ascale, mscale). Columns: `t,gx,gy,gz,ax,ay,az` (minimum for horizon lock). Values as fixed-point integers. Orientation string defines IMU→camera mapping.
- **Gyroflow Synchronization doc** (docs.gyroflow.xyz): For external gyro data, offset can be arbitrary (time between starting gyro recording and video). Autosync algorithm matches optical flow motion to gyro data. Sync points must be consistent (similar offsets). Bad: `0.5ms, -100ms, 20ms`. Good: `-1400ms, -1405ms, -1407ms`.
- **RunCam documentation** (docs.gyroflow.xyz/supported-cameras/runcam): "Correct synchronization of RunCam cameras can be a pain sometimes so if you want to save yourself trouble and frustration, avoid RunCam cameras and get something else instead." Known issues: duplicated/dropped frames, unreliable sync point colors. This is from Gyroflow's official docs — a strong warning.
- **Oscar Liang Gyroflow Tutorial** (oscarliang.com/gyroflow/): Comprehensive guide. Covers using Betaflight blackbox data as external gyro source. The blackbox→Gyroflow workflow is proven — Avatar's ESP32→Gyroflow workflow is equivalent.
- **Gyroflow v1.6.3** (gyroflow.xyz, 2026): Latest release. "Basic stabilization without gyro data" using visual feature tracking as fallback. OpenFX plugin support. This means even without perfect gyro sync, Gyroflow can stabilize using pure optical flow from video — lower quality but functional.

### .gcsv Implementation for ESP32

```python
# Python on MacBook (or C on ESP32 — format is trivial)
# Generate .gcsv from ESP32 ICM-42688 telemetry

GYRO_SCALE = 2000 * 3.14159 / 180 / 32768  # ±2000dps → rad/s per LSB
ACCEL_SCALE = 16 / 32768                     # ±16g → g per LSB
TS_SCALE = 0.001                             # 1kHz sampling = 1ms per tick

gcsv_header = f"""GYROFLOW IMU LOG
version,1.3
id,avatar_esp32_icm42688
orientation,YxZ
note,ESP32 ICM-42688 gyro data for Gyroflow
fwversion,avatar_0.1.0
timestamp,{int(time.time())}
vendor,avatar
tscale,{TS_SCALE}
gscale,{GYRO_SCALE}
ascale,{ACCEL_SCALE}
t,gx,gy,gz,ax,ay,az
"""
# Then append: timestamp_ms, gx_raw, gy_raw, gz_raw, ax_raw, ay_raw, az_raw
```

**Weight of .gcsv file:** At 1kHz, 6 columns (gx,gy,gz,ax,ay,az), 16-bit integers (~4 chars each): ~24 bytes/line × 1000 lines/s × 60s × 5min = ~7.2MB per 5-minute flight. Trivial for the MacBook to store.

### Sync Strategy for Avatar

1. **Pre-flight:** Start ESP32 gyro logging. Start RunCam recording. Flash a bright LED (connected to ESP32 GPIO) 3 times. This creates a visible pattern in both video and a known timestamp in the gyro log.
2. **Post-flight:** Copy `.gcsv` from ESP32 (via WiFi/SD card) and video from RunCam SD card to MacBook.
3. **Gyroflow:** Load video, load `.gcsv` as external gyro source. Set rough gyro offset to the LED flash timestamp. Run autosync on a section with significant motion. Verify with playback loop.
4. **Fallback:** If sync fails, use Gyroflow v1.6.3's visual-only stabilization (no gyro data needed). Quality is lower but functional.
5. **Alternative camera:** Consider the Hawkeye Thumb 4K (listed in facts.md at $60, 16.5g) — it may have better gyro data quality than RunCam Thumb Pro. Test both cameras' `.gcsv` output with Gyroflow before committing.

### Relevance to Avatar

The facts.md notes: "ESP32 ICM-42688 gyro data for Gyroflow sync means any camera works." This finding validates that approach, with the important caveat that RunCam cameras have known sync issues. **The ICM-42688 gyro data route is more reliable than the Thumb Pro's internal gyro.** Avatar should plan for external gyro sync from day one, not as a backup.

- **NOW:** Write the `.gcsv` generator — either Python on MacBook (receiving ESP32 telemetry) or C on ESP32 (writing to SD card). Python is simpler for V1. Test with a sample IMU recording and verify Gyroflow loads it.
